# paper-zh-explainer

把英文论文 PDF 消化成一篇中文技术解读文章，并保存到 `~/NeuralVault/paper-zh-explainer/`。

产物分两大块：第一块是压缩后的中文解读，讲清论文领域、问题、方法、实验和效果；第二块是整篇论文的逐段中文翻译，按原文顺序保留关键图和表格，公式直接写成 Markdown 数学式。全文用作者风格的大白话，情绪嗨、正能量。目标是让读者先快速读懂论文主旨，再能顺着完整翻译追原文细节。

不把产物写入当前 `cat-skill` 仓库；除生成目标 Markdown 和提交 NeuralVault 外，不改动其他文件。

## 依赖资料

生成前读取：

- `../_assets/voice-profile-blog.md`：写作风格硬约束

可选参考：

- `../_assets/1.md`：论文解读样稿

## 输入

```yaml
paper_pdf: <必填，英文论文 PDF 本地路径或 URL>
topic_name: <可选，技术名或模型名；不填则从论文中提取>
cover_text: <可选，即梦封面「名称功能」文案；不填则从论文提炼，格式见 Step 6>
code_url: <可选，用户提供或论文明确给出的代码地址>
focus: <可选，用户特别关心的角度，例如工程实现、模型结构、实验、局限性>
```

如果 `paper_pdf` 不是 PDF、不能读取，或论文关键信息缺失到无法判断主旨，停下要求用户补充材料。

## 工作流

复制并跟踪：

```markdown
Task Progress:
- [ ] Step 1: 读取论文和 voice profile，确认 PDF 可读
- [ ] Step 2: 梳理论文正文、章节和图表公式清单
- [ ] Step 3: 抽取论文元信息卡，不编造缺失信息
- [ ] Step 4: 生成第一大块：论文解读
- [ ] Step 5: 生成第二大块：整篇逐段翻译，公式写 Markdown，保留图和表格
- [ ] Step 6: 生成即梦配图 prompt 和完整 Markdown 文稿
- [ ] Step 7: 写入 NeuralVault
- [ ] Step 8: 自检并修正
- [ ] Step 9: 提交并推送 NeuralVault
```

### Step 1: 读取论文和 voice profile

读取 `paper_pdf` 和 `../_assets/voice-profile-blog.md`。

论文是 PDF 时，优先使用可读取文本；如果 PDF 解析出来的正文明显缺页、乱码或公式/图表严重缺失，说明风险并要求用户补充论文文本或关键截图。

### Step 2: 梳理论文正文、章节和图表公式清单

先梳理可用材料，不要直接开写：

```markdown
- 论文正文解析状态：
- 章节列表：
- 关键 figure / table：
- 关键公式：
```

硬约束：

- `论文正文解析状态` 记录 PDF 是否完整可读；如果缺页、乱码或公式/图表严重缺失，先说明风险。
- `章节列表` 按原文顺序记录标题，供第二大块逐段翻译使用。
- `关键 figure / table` 分两类记录：第一大块只选最关键的一张图；第二大块按原文顺序保留需要出现的图和表。
- `关键公式` 记录公式编号、所在章节、公式在讲什么，以及拟写入文稿的 Markdown 数学式草稿；PDF 解析不可靠时先尝试从上下文还原，仍无法还原则标注并请用户补充，不要用 `【equation X】` 占位。

### Step 3: 抽取论文元信息卡

只抽元信息，不在这一步判断论文贡献。

```markdown
- 论文原名：
- 作者：
- 团队 / 机构中文名：
- 发表时间：
- 代码地址：
```

硬约束：

- 论文原名、作者、团队、发表时间、代码地址必须来自论文或用户材料。
- 作者不要全量罗列。挑 1-3 个最值得介绍的代表作者，说明其机构、代表工作或为什么值得读者关注。
- 如果不能确定作者履历或代表工作，只写姓名即可，不要硬编"大神"背景。
- 团队 / 机构用中文常用名，例如斯坦福、加州大学伯克利分校、丰田研究院、Google DeepMind、MIT。
- 论文没给、PDF 没读到、信息不确定时，写"论文中未明确给出"或停下问用户。
- 不为了文章完整而编具体时间、数字、比例、代码地址或团队归属。

### Step 4: 生成第一大块：论文解读

第一大块只做浓缩解读，不逐段翻译。必须按下面顺序讲清楚：

1. 论文是什么领域的。
2. 这篇论文为了解决什么问题。
3. 作者怎么解决这个问题。
4. 如有必要，只贴最关键的一张图，并用大白话解释它在说明什么。
5. 作者用了什么实验方法验证。
6. 实验效果如何，哪些结果最能支撑论文贡献。
7. 这篇论文的限制和工程使用判断。

判断规则：

- 领域要具体到读者能定位论文位置，例如机器人操作、视觉语言模型、强化学习、生成式世界模型，不要只写"人工智能"。
- 问题要比论文标题更具体，指出作者试图打掉的关键障碍。
- 方法要讲清作者怎么绕开旧做法的限制，而不只是列模块名。
- 实验方法要讲清作者怎么验证：数据集 / benchmark / 对比基线 / 消融 / 真实系统测试，能写具体就写具体。
- 实验效果必须来自实验、消融、图表、作者论述或明确案例。
- 最关键配图最多选一张。只有这张图能显著降低理解成本时才贴；没有必要就不贴。
- 局限和工程使用判断要诚实，但用大白话讲，别写成冷冰冰的审稿意见。
- 如果无法用 3-5 句说清楚这篇论文为什么值得读，先回到论文重读摘要、引言、方法总览、实验结论和局限性，不要硬写正文。
- 全文基调：作者风格的大白话，情绪又嗨又正能量——让读者觉得「这论文有意思、值得看」，同时判断仍来自论文事实。

### Step 5: 生成第二大块：逐段翻译全文

第二大块按原文顺序逐段翻译整篇论文，不重组结构。

规则：

- 保留原论文一级 / 二级标题，并翻译成中文；必要时在标题后保留英文原名。
- 每个自然段都要翻译，不能只摘要。
- 不要逐句硬译；以自然段为单位译成顺畅中文，但不得删掉原文信息。
- 图、表格按原文出现位置保留 `【figure X】` / `【table X】` 占位，并在占位后用大白话说明。
- 公式**不留占位**：在公式首次出现的段落，先用大白话讲它在干嘛，再用 Markdown 数学式写出（行内 `$...$`，独立一行用 `$$...$$`）；符号与论文一致，编号可写在公式后括号里，例如 `(式 3)`。
- 表格用 Markdown 表格能可靠表达时转成 Markdown；复杂表格保留 `【table X】` 占位并说明表格含义。
- 如果 PDF 的某个段落、公式、图表无法可靠解析，标注"此处 PDF 解析不完整"，不要自行补内容。

### 图表占位与公式写法

**图 / 表**：需要插入论文图、表或结构图的位置，单独留一行占位：

```markdown
【figure X】
```

规则：

- `X` 使用论文原始编号，例如 `【figure 1】`、`【figure 3】`、`【figure 4a】`；表格用 `【table X】`。
- 占位符前后各空一行。
- 第一大块最多只贴一张最关键图；第二大块按原文顺序保留全文翻译需要的图和表。
- 占位符后面要用大白话解释这张图/表想说明什么。

**公式**：不使用 `【equation X】`。在解读或翻译中写到该公式时，于该段落后直接嵌入 Markdown 数学式，例如：

```markdown
损失可以看成「预测错了多少」的平方和：

$$
\mathcal{L} = \frac{1}{N}\sum_{i=1}^{N}(y_i - \hat{y}_i)^2 \quad \text{(式 2)}
$$
```

**专有名词补充**（每篇只做 1-2 个）：

- 在梳理 Step 2 时标出全文最关键的 1-2 个专有名词（读者最可能卡住的，例如「湿实验」「world model」）。
- 在该词**第一次出现的段落末尾**，另起一句用大白话补充含义，语气与正文一致，可带一点嗨感，例如：「湿实验」就是真在实验室里动手做、不是纯仿真跑出来的那种实验。
- 不要给每个术语都加注释；次要术语第一次出现时括号里一句话带过即可。

### Step 6: 生成即梦配图 prompt 和 Markdown 解读文

输出必须是 Markdown。

推荐结构：

```markdown
## 配图 prompt

艺术字设计，“[名称功能]”几个大字，字体为手写风格，字体为无衬线体，粗体，笔画粗细统一，部分笔画进行连笔处理，线条流畅灵动，有飘逸感，字体颜色为白色，黄色装饰。英文“for freedom”在文字旁，颜色为黄色，背景为深色，有些隐约的纹理，整体风格潮流，活力，具有运动氛围感。

---


# [缩写/技术名] 模型解读：[吸睛但准确的重点描述]

论文：[论文原名]

代码：[代码地址；没有就省略]

## 论文元信息

- 发表时间：[时间]
- 代表作者：[挑 1-3 个代表作者，简要说明来历；不要全量列名单]
- 团队：[中文常用机构名]

---

## 一、论文解读

### 1. 这是什么领域的论文

[用作者风格的大白话说明论文所属领域，以及它在这个领域里大概站在哪个问题线上；语气嗨、正能量，别端着。]

### 2. 它想解决什么问题

[基于 Step 4 的判断规则，讲清楚这篇论文真正解决的问题、为什么这个问题难。]

### 3. 它是怎么解决的

[用大白话讲清作者实际上用了什么原理什么核心方法和关键转折，不要只堆模块名，可以适当使用比喻；关键公式直接写 Markdown 数学式，不要留 equation 占位。]

【figure X】

[如有必要，只贴最关键的一张图，并解释这张图为什么能帮助理解方法；没有必要就删掉这一段。]

[若本节出现全文首个「关键专有名词」，在该段末尾用一句话补充解释，见「专有名词补充」规则。]

### 4. 实验结果如何

[用大白话讲清，作者使用了什么数据集，控制了什么变量，得到了什么关键结论。]

---

## 二、逐段翻译

### Abstract / 摘要

[逐段翻译摘要。]

### 1. Introduction / 引言

[逐段翻译引言。]

【figure X】

[解释这张图说明了什么。]

[遇到公式：先大白话讲它在干嘛，再写 Markdown 数学式，不要用 【equation X】。]

【table X】

[解释这个表格说明了什么。]

### 2. [原论文标题中文翻译] / [Original Section Title]

[继续按原文顺序逐段翻译。]
```

### Step 7: 写入 NeuralVault

将 Markdown 写入：

```text
~/NeuralVault/paper-zh-explainer/
```

文件名使用：

```text
YYYY-MM-DD-[技术名或论文短名]-解读.md
```

规则：

- 目标目录不存在时先创建。
- 文件名用中文或英文都可以，但不要包含 Windows 非法文件名字符。
- 如果同名文件已存在，先读取旧文件；确认是同一篇论文的旧版本后覆盖，否则换一个更具体的文件名。

### Step 8: 自检并修正

写作要求：

- `## 配图 prompt` 必须放在文章开头、正文标题之前。
- 配图 prompt 与 `wechat-publish-package` 使用同一套固定文案骨架，只替换 `[名称功能]`，不要改写风格词。
- `[名称功能]` = **技术名/模型名 + 一句话功能**，让人一眼知道「谁、干了啥」。示例：`BioProAgent给湿实验Agent装上刹车`。优先用用户提供的 `cover_text`；否则从论文方法里提炼，一般不超过 20 个字。
- **语气硬约束**：全文用 `voice-profile-blog.md` 里的大白话，情绪又嗨又正能量——教学段落也要带劲，别写成说明书；评价、亮点、收束可以更放飞（模式 B），但别把论文解读写成纯段子。
- 第一大块 `## 一、论文解读` 必须有判断力，判断来自论文事实；可以写「惊艳」「漂亮」「真香」，但要说出具体原因。
- 第一大块必须讲清领域、问题、方法、实验方法、实验效果、局限和工程使用判断。
- 第一大块如需贴图，最多只贴最关键的一张图。
- 第二大块 `## 二、逐段翻译` 必须按原文顺序逐段翻译整篇论文，不能改写成摘要。
- 第二大块必须按原文位置保留需要出现的图和表格；公式用 Markdown 数学式嵌入，不用 `【equation X】`。
- 每篇只对最关键的 1-2 个专有名词做「段末补充解释」；其余术语第一次出现括号带过即可。
- 复杂公式不要硬翻成人话公式，先讲公式在干嘛，再写 Markdown 数学式保留必要符号。
- 实验数字尽量保留具体值；没读到具体数字就不要补。
- 段落偏短，长论证后接一句短总结。

输出前逐项检查：

- [ ] 是 Markdown，不是纯文本。
- [ ] 文章开头有 `## 配图 prompt`，且位于正文标题之前。
- [ ] 配图 prompt 固定文案一致，封面大字为「名称功能」格式（技术名 + 一句话功能）。
- [ ] 全文大白话，情绪嗨、正能量，不是冷冰冰说明书腔。
- [ ] 标题包含技术名/缩写和一个准确重点。
- [ ] 已先完成论文元信息卡，再开始写第一大块。
- [ ] 文稿包含 `## 一、论文解读` 和 `## 二、逐段翻译` 两大块。
- [ ] 第一大块讲清楚论文所属领域、要解决的问题、解决方法、实验方法、实验效果、局限和工程使用判断。
- [ ] 第一大块最多只贴一张最关键图；如果贴了，图后有解释。
- [ ] 开头的价值判断都有论文事实支撑，没有空泛吹捧。
- [ ] 论文原名、发表时间、代表作者、团队没有编造。
- [ ] 作者没有全量罗列；只介绍 1-3 个有辨识度的代表作者。
- [ ] 团队 / 机构使用中文常用名。
- [ ] 第二大块按原文顺序逐段翻译整篇论文，没有只翻摘要或只做概括。
- [ ] 第二大块图/表位置保留了 `【figure X】` 或 `【table X】`；公式已写成 Markdown 数学式，无 `【equation X】`。
- [ ] 每个图/表占位后都有解释；关键公式所在段有「讲人话 + 数学式」。
- [ ] 全文仅对 1-2 个最关键专有名词做了段末补充解释。
- [ ] 遵守 `voice-profile-blog.md`，没有出现黑名单词。
- [ ] 不确定事实已标注或已向用户提问。
- [ ] Markdown 已写入 `~/NeuralVault/paper-zh-explainer/`。

任一项不满足，先修正文稿，再交付。

### Step 9: 提交并推送 NeuralVault

每次解读完成后，对 `~/NeuralVault` 仓库执行一次 git commit 和 push。

提交规则：

- 只提交本次生成或更新的论文解读 Markdown。
- 提交前查看 `git status`，确认没有把无关文件一起提交。
- commit message 使用：

```text
add [技术名或论文短名] paper explainer
```

- commit 成功后执行 `git push`。
- 如果 `~/NeuralVault` 不是 git 仓库、没有远端、commit 或 push 失败，停止并把失败原因告诉用户，不要在当前仓库补交。

## 验证本 skill 是否起效

- 生成的 Markdown 位于 `~/NeuralVault/paper-zh-explainer/`。
- 文稿符合自检清单。
- NeuralVault 仓库产生了只包含本次解读文件的 commit。
- commit 已成功 push 到远端；如果失败，已明确说明失败原因。

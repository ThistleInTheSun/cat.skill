# paper-zh-explainer

把英文论文 PDF 消化成一篇中文技术解读文章，并保存到 `~/NeuralVault/paper-zh-explainer/`。

这不是逐句翻译。目标是让读者不看英文论文，也能知道这篇论文解决了什么问题、用了什么办法、实验说明了什么、局限在哪里，以及这东西跟工程或应用有什么关系。

不把产物写入当前 `cat-skill` 仓库；除生成目标 Markdown 和提交 NeuralVault 外，不改动其他文件。

## 依赖资料

生成前读取：

- `../_assets/voice-profile-blog.md`：写作风格硬约束

可选参考：

- `../_assets/1.md`：论文解读样稿

## 输入

```yaml
paper_pdf: <必填，英文论文 PDF 本地路径或 URL>
topic_name: <可选，技术名或模型名；不填则从论文中提取>
cover_text: <可选，即梦封面大字，不超过 10 个字；不填则从技术名和标题提炼>
code_url: <可选，用户提供或论文明确给出的代码地址>
focus: <可选，用户特别关心的角度，例如工程实现、模型结构、实验、局限性>
```

如果 `paper_pdf` 不是 PDF、不能读取，或论文关键信息缺失到无法判断主旨，停下要求用户补充材料。

## 工作流

复制并跟踪：

```markdown
Task Progress:
- [ ] Step 1: 读取论文和 voice profile
- [ ] Step 2: 抽取论文事实卡
- [ ] Step 3: 建立正文逻辑块
- [ ] Step 4: 标记需要插图的位置
- [ ] Step 5: 生成即梦配图 prompt 和 Markdown 解读文
- [ ] Step 6: 写入 NeuralVault
- [ ] Step 7: 自检并修正
- [ ] Step 8: 提交并推送 NeuralVault
```

### Step 1: 读取论文和 voice profile

读取 `paper_pdf` 和 `../_assets/voice-profile-blog.md`。

论文是 PDF 时，优先使用可读取文本；如果 PDF 解析出来的正文明显缺页、乱码或公式/图表严重缺失，说明风险并要求用户补充论文文本或关键截图。

### Step 2: 抽取论文事实卡

先抽事实，不要直接开写：

```markdown
- 技术名 / 缩写：
- 论文原名：
- 发表时间：
- 代表作者：
- 团队 / 机构中文名：
- 代码地址：
- 核心任务：
- 核心方法：
- 数据 / 训练规模：
- 主要实验结论：
- 局限性：
- 关键 figure / table：
```

硬约束：

- 发表时间、作者、团队、数据规模、实验数字、代码地址必须来自论文或用户材料。
- 作者不要全量罗列。挑 1-3 个最值得介绍的代表作者，说明其机构、代表工作或为什么值得读者关注。
- 如果不能确定作者履历或代表工作，只写姓名和论文中给出的机构，不要硬编"大神"背景。
- 团队 / 机构用中文常用名，例如斯坦福、加州大学伯克利分校、丰田研究院、Google DeepMind、MIT。
- 论文没给、PDF 没读到、信息不确定时，写"论文中未明确给出"或停下问用户。
- 不为了文章完整而编具体时间、数字、比例或团队归属。

### Step 3: 建立正文逻辑块

按论文逻辑块解读，不按 PDF 自然段逐段翻译。

常见逻辑块：

- 动机：作者为什么要做这件事
- 方法：模型/算法/系统到底怎么做
- 训练或数据：数据从哪来，训练怎么搞
- 实验：实验想证明什么，结果说明什么
- 对比：跟已有方法比，赢在哪，输在哪
- 局限性：作者自己承认了什么问题
- 工程使用：如果论文给了代码、模型或上手方式，就讲怎么用

如果论文结构和上面不一致，按论文自己的逻辑重组。

### Step 4: 标记需要插图的位置

需要插入论文图、表或结构图的位置，单独留一行占位：

```markdown
【figure X】
```

规则：

- `X` 使用论文原始编号，例如 `【figure 1】`、`【figure 3】`、`【figure 4a】`。
- 如果是表格，用 `【table X】`。
- 占位符前后各空一行。
- 只有图表对理解有帮助时才插入，不要为了形式把所有图都塞进来。
- 占位符后面要用大白话解释这张图/表想说明什么。

### Step 5: 生成即梦配图 prompt 和 Markdown 解读文

输出必须是 Markdown。

推荐结构：

```markdown
## 配图 prompt

艺术字设计，“[封面短标题]”几个大字，字体为手写风格，字体为无衬线体，粗体，笔画粗细统一，部分笔画进行连笔处理，线条流畅灵动，有飘逸感，字体颜色为白色，黄色装饰。英文“for freedom”在文字旁，颜色为黄色，背景为深色，有些隐约的纹理，整体风格潮流，活力，具有运动氛围感。

---

# [缩写/技术名] 模型解读：[吸睛但准确的重点描述]

[一句大白话说清楚这项技术是干嘛的。]

论文：[论文原名]

代码：[代码地址；没有就省略]

## 论文身份卡

- 发表时间：[时间]
- 代表作者：[挑 1-3 个代表作者，简要说明来历；不要全量列名单]
- 团队：[中文常用机构名]
- 任务：[核心任务]

## 极简原理

[用一小段或几条讲清楚核心做法。]

---

## 一、[逻辑块标题]

[用大白话解释作者想表达什么。]

【figure X】

[解释这张图/表说明了什么。]

## 二、[逻辑块标题]

[继续按逻辑块解读。]

## 局限性

[如果论文有局限性，客观列出；没有就省略。]

## 快速上手

[如果论文或代码给了使用方式，写安装/调用/入口；没有就省略。]
```

### Step 6: 写入 NeuralVault

将 Markdown 写入：

```text
~/NeuralVault/paper-zh-explainer/
```

文件名使用：

```text
YYYY-MM-DD-[技术名或论文短名]-解读.md
```

规则：

- 目标目录不存在时先创建。
- 文件名用中文或英文都可以，但不要包含 Windows 非法文件名字符。
- 如果同名文件已存在，先读取旧文件；确认是同一篇论文的旧版本后覆盖，否则换一个更具体的文件名。

### Step 7: 自检并修正

写作要求：

- `## 配图 prompt` 必须放在文章开头、正文标题之前。
- 配图 prompt 与 `wechat-publish-package` 使用同一套固定文案，只替换 `[封面短标题]`，不要改写风格词。
- 封面短标题不超过 10 个字，优先保留技术名和最有记忆点的词。
- 技术解读默认使用 voice profile 的模式 A，整体平实，把事讲清楚。
- 遇到评价、吐槽、总结时，可以短暂切到模式 B，但不要把论文解读写成段子。
- 术语第一次出现时，先给英文，再用大白话解释。
- 复杂公式不要硬翻，先讲公式在干嘛，再保留必要符号。
- 实验数字尽量保留具体值；没读到具体数字就不要补。
- 段落偏短，长论证后接一句短总结。

输出前逐项检查：

- [ ] 是 Markdown，不是纯文本。
- [ ] 文章开头有 `## 配图 prompt`，且位于正文标题之前。
- [ ] 配图 prompt 与 `wechat-publish-package` 的固定文案一致，只替换封面短标题。
- [ ] 封面短标题不超过 10 个字。
- [ ] 标题包含技术名/缩写和一个准确重点。
- [ ] 开头一句话讲清楚技术用途。
- [ ] 论文原名、发表时间、代表作者、团队没有编造。
- [ ] 作者没有全量罗列；只介绍 1-3 个有辨识度的代表作者。
- [ ] 团队 / 机构使用中文常用名。
- [ ] 正文按逻辑块解读，不是逐句翻译。
- [ ] 需要插图的位置保留了 `【figure X】` 或 `【table X】`。
- [ ] 每个图表占位后都有解释。
- [ ] 遵守 `voice-profile-blog.md`，没有出现黑名单词。
- [ ] 不确定事实已标注或已向用户提问。
- [ ] Markdown 已写入 `~/NeuralVault/paper-zh-explainer/`。

任一项不满足，先修正文稿，再交付。

### Step 8: 提交并推送 NeuralVault

每次解读完成后，对 `~/NeuralVault` 仓库执行一次 git commit 和 push。

提交规则：

- 只提交本次生成或更新的论文解读 Markdown。
- 提交前查看 `git status`，确认没有把无关文件一起提交。
- commit message 使用：

```text
add [技术名或论文短名] paper explainer
```

- commit 成功后执行 `git push`。
- 如果 `~/NeuralVault` 不是 git 仓库、没有远端、commit 或 push 失败，停止并把失败原因告诉用户，不要在当前仓库补交。

## 验证本 skill 是否起效

- 生成的 Markdown 位于 `~/NeuralVault/paper-zh-explainer/`。
- 文稿符合自检清单。
- NeuralVault 仓库产生了只包含本次解读文件的 commit。
- commit 已成功 push 到远端；如果失败，已明确说明失败原因。

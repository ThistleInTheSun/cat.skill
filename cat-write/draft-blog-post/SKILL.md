# draft-blog-post

为用户在 `~/NeuralVault/drafts/` 下生成一篇公众号/博客文章的粗稿（v1）。

## 定位

类比写小说的工序：本 skill 是「先把骨架填出血肉」那一步，**只出第一稿**，且**故意中等质量**——目标是「让用户改起来比从头写舒服」，不是「一稿到位的成品」。

## 边界

- ✅ 做：基于用户的 thesis + bones，按 voice profile 写出 800-1500 字粗稿
- ❌ 不做：不修改既有草稿（那是未来 `paragraph-triage` 的事）
- ❌ 不做：不做整篇声音迁移（那是未来 `voice-rewrite` 的事）
- ❌ 不做：不做发布前检查（那是未来 `pre-publish-checklist` 的事）
- ❌ 不做：不查任何外部资料、不编造引用、不编造数据
- ❌ 不做：thesis 缺失或 bones 少于 3 条时**不开干**，必须先问用户补

## 为什么强制要求 thesis + bones

完美主义型作者最大的崩点是：「AI 写的不是我想说的」。

让 AI 从主题自由发挥 → 它会编一套自己的论点 → 用户改的时候发现骨架就是错的 → 等于重写 → 信心崩盘。

**强制 thesis + bones 是把"思想"留给用户、把"扩展词句"交给 AI**——用户心理上能认领「这是我的文章被润色」，而不是「这是 AI 的文章我在改」。

这一条是本 skill 的核心约束。不允许妥协。

## 输入

```yaml
topic:      <一句话主题，必填>
thesis:     <核心观点 1-3 句，必填>
bones:      <论证骨架 3-5 条 bullet，每条 = 一个分论点 / 一个具体例子 / 一个反例，必填>
mode:       <可选，A | B | mixed；不填则由 AI 推断并在回复中告知用户>
audience:   <可选，默认 "公众号读者">
length:     <可选，默认 1000 字，区间 800-1500>
notes:      <可选，特别要避免/包含什么>
```

模式含义（详见 voice-profile-blog.md 的"模式分档"段）：

- **A**：教学 / 陈述（技术解读、概念翻译、流程说明、知识科普）。平实为主，不强行加皮
- **B**：观点 / 评价（鸡汤、职场吐槽、经验分享、感慨总结）。放飞，带情绪
- **mixed**：A 为主 + 偶尔 B 段穿插（最常见）

### 输入校验

进入工作流前先检查：

- [ ] `thesis` 非空，且不只是复述 `topic`
- [ ] `bones` ≥ 3 条
- [ ] 每条 bone ≥ 5 个字，不是空泛标签（如「重要性」「分析」一类零信息词）

任一不满足 → **停下，回复用户**：

```
缺少必要输入，无法开工：
- thesis：[当前状态 / 缺失 / 太空泛]
- bones：[N 条，需要 ≥ 3 条；当前过空的是：...]

请先补齐。提示：bones 不是大纲标题，是你脑子里已经有的"我想说什么、用什么例子说"。
如果你 5 分钟列不出 3 条 bones，说明这个主题你还没想清楚，先放一放。
```

## 必读资产

每次生成前必须读取并遵守：

- `voice-profile-blog.md`（与本 SKILL.md 同目录）

资产中的句式 DO/DON'T、词汇白/黑名单、修辞偏好、结构模板、自检清单，**全部当作硬约束**。

## 工作流

复制并跟踪：

```
- [ ] 1. 校验输入（thesis + bones）
- [ ] 2. 读取 voice-profile-blog.md
- [ ] 3. 识别模式（整篇模式 + 每条 bone 的模式标签）
- [ ] 4. 规划结构（按 voice 默认结构模板映射每个 bone）
- [ ] 5. 生成正文（每段按其模式标签调用对应规则）
- [ ] 6. 套用开篇 + 「写在最后」收束
- [ ] 7. 跑 voice 自检清单（按整篇模式选用 A / B 分组）+ 本 skill 自检
- [ ] 8. 写入 ~/NeuralVault/drafts/<YYYY-MM-DD>-<slug>.md
- [ ] 9. 简短回复用户：路径 + 字数 + 模式判定 + 自检结果 + 下一步建议
```

### Step 3 — 识别模式

如果用户在输入里给了 `mode`，直接用。否则按下面推断：

1. **整篇模式判定**：基于 topic + thesis 的性质
   - topic / thesis 是"讲清一件事 / 一个概念 / 一个原理" → 整篇 **A**
   - topic / thesis 是"我觉得 / 你应该 / 怎么办" → 整篇 **B**
   - topic 是讲事实但 thesis 带主观倾向 → **mixed**（A 主体）

2. **每条 bone 标模式标签**（对照 voice profile 的判定矩阵）：
   - 客观陈述、概念翻译、流程、数据 → bone[A]
   - 个人吐槽 / 感慨 / 经验 / 鸡汤 → bone[B]

3. **不确定时默认 A**——保守为主，避免硬塞情绪。

把判定结果记下来，输出时要告诉用户。

### Step 4 — 结构规划

把 bones 映射到 voice 的默认结构：

| voice 结构段 | 内容来源 | 默认模式 |
|--------------|----------|---------|
| 标题 | 从 topic 提炼，可带情绪 / 态度 | 跟整篇 |
| 开头 1-3 句 | thesis + 自我定位 / 读者圈定 | 跟整篇 |
| 动机段 | 「为什么写这个」；纯技术文可省略 | 整篇 A 时可省；整篇 B 时必有 |
| 核心展开 | 每个 bone → 1 段，必须编号或加小标题 | 按 bone 标签 |
| 局限 / 反例段 | 若 bones 里有反例 bone 用它；否则自己加一段「当然，也有例外...」 | 跟整篇 |
| 写在最后 | 一句鼓舞 / 一个行动 / 一句自嘲 | 永远 B（即使整篇是 A） |

### Step 5 — 扩写规则

每段写作时：

- **先看本段模式标签**，调用 voice profile 中对应的 DO / DON'T 与白名单
  - bone[A] 段：平实，不强塞 `！！！` / `~~~` / `「狗头」`；优先用大白话翻译 + 数据 + 类比
  - bone[B] 段：放飞，反问 / 重复句 / 自身经验 / 括号自嘲
- 必须用 voice 修辞偏好里的至少 1 种（场景代入 / 对比 / 数据 / 类比 / 自嘲）
- 不允许引入 bones 里没有的"新论点"——只能扩展和润色
- 不允许编数据、编人名、编引用；若 bone 里有数字就用 bone 里的数字
- 句长控制：一段内不要有超过 3 个嵌套从句的长句
- **整篇 A 时**：B 模式句子穿插不超过全文 20%，且只出现在评价 / 感慨 / 收束位置
- 正文不要使用 Markdown 无序列表（`-` / `*` / `+`）；需要分点时，改成有序列表或「第一点 / 第二点」这类行文表达

## 产物

### 文件路径

```
~/NeuralVault/drafts/<YYYY-MM-DD>-<slug>.md
```

- `<YYYY-MM-DD>` = 当天真实日期
- `<slug>` = 从 topic 转化的 lowercase + hyphens，最多 6 个词；中文 topic 可保留中文，去除标点替换为 `-`

例：
- topic「为什么 code review 不该追求覆盖率」 → `2026-05-19-为什么-code-review-不该追求覆盖率.md`
- topic「Why code review should not chase coverage」 → `2026-05-19-code-review-coverage.md`

### 文件内容格式

```markdown
---
title: <标题>
thesis: <一句话复述 thesis>
status: draft-v1
created: YYYY-MM-DD
voice: blog
word_count: <实际字数>
---

# <标题>

<正文>
```

## 自检清单（输出前必须全部通过）

逐项核对，任一失败 → 重写后再输出：

### 通用必过

- [ ] 字数在 800-1500
- [ ] thesis 在文章前 20% 内被明确表达（意思到位即可，不必原文复述）
- [ ] 每条 bone 都有对应的正文段，无遗漏
- [ ] 没有引入 bones 之外的"新论点"
- [ ] 没有编造任何数据、人名、引用
- [ ] 通过 voice profile 的「全模式必过」自检
- [ ] 末尾有「写在最后」或等价的收束段（不论整篇是 A 还是 B，收束都用 B）
- [ ] 没有出现 voice 词汇黑名单中的任何词（赋能 / 加持 / 颠覆 / 综上所述 / ...）

### 按整篇模式分组

- 整篇 A → 必过 voice profile 的「模式 A 主导的文章」自检
- 整篇 B → 必过 voice profile 的「模式 B 主导的文章」自检
- mixed → 整篇按 A 自检；同时核对 B 句穿插是否 ≤ 20% 且只在评价 / 感慨位置

## 输出后给用户的回复模板

```
草稿已生成：~/NeuralVault/drafts/<...>.md
字数：<N>
模式判定：<A / B / mixed (主 A，B 段：第 X、Y 段)>
voice 自检：通过 / 失败项 [...]
bones 覆盖：<M>/<总数> 全部覆盖

下一步建议：
1. 通读一遍，挑出 1-3 个"我绝对要重写"的段落 + 1-3 个"还行可保留"的段落
2. 把这两类段落贴回来，我们用真实反馈设计 paragraph-triage skill
3. 不要追求一稿到位——v1 故意中等质量，能改才是目的

如果模式判定错了 → 告诉我应该是哪种，下次重写时强制指定 mode 字段。
如果你读完的感觉是「还不如我自己写」，先排查两个地方：
  - bones 是不是太空？AI 没素材就只能水
  - voice profile 哪条不匹配？告诉我具体哪句话不像你
```

## 反例

- 🔴 用户只给主题没给 thesis 和 bones，AI 替用户编 → 必须停下问，不开干
- 🔴 写完顺手再"优化润色"几遍 → 越界，那是下游 skill 的事
- 🔴 引用了具体数据/案例但用户没给 → 编造，禁止
- 🔴 出现 voice 黑名单词：「这个方法极具颠覆性」「综上所述」🔴
- 🔴 写成议论文八股：「首先...其次...再次...最后...」🔴
- 🔴 字数超过 1500「为了详尽」🔴 强行砍
- 🔴 把 voice profile 当"参考"而不是"硬约束"，自由发挥 🔴
- 🔴 bone 是「重要性」「价值」「分析」等空泛标签，AI 还是开工 🔴 必须先打回
- 🔴 bone 没给具体时间 / 次数 / 比例，AI 自己编了具体数字（如"一个星期"、"反复 5 次"、"准确率 30%"）🔴 用模糊词（"反复折腾"、"好一阵子"、"明显下降"）或反问用户「这里的时间 / 数量你有具体数字吗？」

## 如何判断本 skill 起效

执行后产物应满足：

- 文件存在于 `~/NeuralVault/drafts/` 下，命名符合 `<YYYY-MM-DD>-<slug>.md`
- 字数在 800-1500
- frontmatter 完整（title / thesis / status / created / voice / word_count）
- 每条 bone 在正文都能找到对应段落
- voice 自检清单全过

**核心成功指标**：用户读完粗稿的真实感受 = 「改起来比从头写舒服」。

失败信号：
- 用户说「还不如自己写」→ 排查 bones 是否过空，或 voice profile 是否漏抓某种你的语感
- 用户说「这不是我想说的」→ thesis 输入校验放得太松，AI 跑偏了
- 用户说「读着像 AI」→ voice 自检没真正执行

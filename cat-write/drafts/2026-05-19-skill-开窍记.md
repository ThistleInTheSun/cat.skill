# 折腾了一个月，我终于在 skill 上开窍了

折腾了一个月，几十个 skill 写下来，我才终于搞明白一件事：

agent+skill 不是神，它本质上就是一个函数。

就这么短一句话。但中间走的弯路，能填一条沟。

## 你是不是也这样

你是不是也在干这种事：看到网上有人晒一个 skill 就能做出精美作品，立刻心动，自己也开干。

写了一打 skill，每个都觉得"这次应该可以了"。结果用起来，要么 AI 答非所问，要么自己看完输出还是得重新动手改。

慢慢就开始怀疑：是不是我不够聪明？是不是我的模型版本不行？是不是网上的大神有什么秘密武器？

都不是。是你（也是我）从一开始，诉求就没说清楚。

## skill 其实就是一个函数

想清楚这件事之后，我才开始能写出真正能用的 skill。

skill 就是用自然语言写的函数。函数有什么？四件事必须定义清楚：

1）**适用条件**：什么情况下用、什么情况下不用。不写清楚，agent 见了就调，结果驴唇不对马嘴。

2）**输入**：必须给什么字段、什么格式。缺了直接拒绝，别替用户编。

3）**输出**：长什么样、怎么算合格。没有评判标准，模型就发挥自由意志。

4）**处理过程的原则**：不是死步骤，是原则——遇到边界情况按什么准则判。

把这四件事写明白，skill 才算是个函数，而不是一段许愿。

## skill 的 5 个级别

自驾领域有 L1-L5，skill 我也按这个分一下，方便定位自己写到哪了：

**L1** 是笼统的"你是谁、你负责干啥"，输出全靠模型心情。

**L2** 是在 L1 基础上加了执行决策树——如果 A 就做 1，如果 B 就做 2，路径清晰了。

**L3** 是在 L2 基础上加了输出标准，跑完自己对照一遍合不合格，不合格就重写。

**L4** 是 description 写得足够精确，agent 不用你提示，自己就能识别什么时候该调你。

**L5** 是多个 skill 自动编排——agent 自己决定调度顺序，把单个小 skill 串成完整工作流，不用你来当指挥。

我目前大部分 skill 卡在 L2-L3 之间，少数到 L4，L5 还在搭路上。

## 别想一口吃成大胖子

想通了 skill 是函数之后，第二个开窍点就是：

**别写大 skill。**

你不会写一个 1000 行的函数对吧？skill 也一样。一个 skill 只做一件事，单一职责。

复杂任务怎么办？拆——拆成一串单一职责的小 skill，再让它们组合调用（这就是 L5 想干的事）。

写得越小，越容易调试、越容易复用、越容易让 agent 准确触发。

## 一个真实反例：我曾经怎么栽的

反例就在我自己身上。

刚开始我想搞个"代码 debug skill"——把整套排查流程一锅炖：分析报错、定位文件、提假设、试方案、回归测试，全塞一个 skill 里。

写完一跑，AI 在某一步走偏了。我开始调 skill。调完再跑，又在另一步崩了。再调，又把前面那段调坏了。

来回一个下午，skill 没调出来，真正要 debug 的代码反而没人管。

最尴尬的是后面——skill 用不顺手，我就不用了，工作直接回退到自己运行程序、自己分析日志再告诉 cursor 怎么修改的迭代状态。**白折腾。**

## 写在最后

所以啊，别再追求那种"一个 skill 解决所有问题"的爽感了。

把每个 skill 当成一个函数，认认真真定义它的适用条件、输入、输出、处理原则——它就能稳稳干活。

把一堆小函数攒起来自动组合——这才是 agent+skill 真正的杀伤力。

开窍这件事一旦发生，整个人都松快了！

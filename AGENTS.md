# AGENTS.md — `cat.skill` 项目 AI 工作指南

> 给在此项目中工作的 AI 看。**开启新窗口先读完本文件**，确保与历史协作脉络无缝衔接。
>
> 本文分两类内容：第 1-5 节是**稳定约定**（项目哲学、用户画像、协作原则），第 6-7 节是**当前进度**（带日期戳，会变）。如果稳定约定与当前进度冲突，以最新日期的进度为准并反过来更新约定。

---

## 1. 项目是什么

`cat.skill` 是用户的个人 skill 仓库——把自己反复使用的隐性知识，外化成可被 AI 反复调用的"技能资产"。目的不是"让 AI 帮忙"，而是**让用户从"演员"变成"导演"**：只在真正需要判断的节点出现，把执行交出去。

### 仓库约定

- 顶层是**大类**，命名 `cat-**`（如 `cat-skill`、`cat-write`）
- 大类下是**具体 skill**，目录必须含 `SKILL.md`
  - `SKILL.md` 顶部用 YAML frontmatter，字段 `name`（lowercase + hyphens，≤ 64 字符）和 `description`（第三人称，含 WHAT + WHEN + 触发词）
- **资产文件**（被 skill 引用但本身不被 agent 触发）放 `<category>/_assets/`
- **产物文件**（草稿、报告等）放 `<category>/drafts/` 或对应位置
- 路径用 `/` 而非 `\`，避免时间敏感措辞（"截至 2024 年..."）

## 2. 用户画像（必须知道的几点）

| 维度 | 关键事实 | 协作含义 |
|------|---------|----------|
| 元思维 | 已经懂 progressive disclosure、单一职责、第三人称 description、PASS/FAIL 二元判定 | 不要给"skill 写作入门"型科普 |
| 写作阻力 | **完美主义型**——核心阻力是"边写边评"的内心拷问，不是技能 / 速度问题 | 写作 skill 必须"分离生成与评估"，让 AI 接管初稿的心理责任 |
| 发布阻力 | **不是"写不好"，是"被关注"**——首篇发布时主动选"不点提醒所有人"（半藏模式） | `pre-publish-checklist` 不能只查内容质量，必须给用户「曝光度分级」选项（草稿 / 半藏 / 推送 / 转发） |
| 写作动机 | **表达本身就是奖励**——"把观点送出去就很开心"，不为涨粉、阅读量、变现 | 写作 skill 的成功指标 = "用户愿意再写一篇"。不要往涨粉 / 标题党 / 商业化方向引 |
| 倾向 | **工程师式基础设施崇拜**——容易花太多时间打磨工具不解决问题 | 不断拉回真实需求；warning 当 meta-skill > 业务 skill 时 |
| 自我认识 | 相当强，能精准诊断自己问题（如知道自己卡在"开始的痛苦"非"做的痛苦"） | 不要预设他不懂；他说"我觉得 X" 的时候 X 通常是对的 |
| 沟通偏好 | 拒绝"看起来正确但没有深度"的回答；中文回复 | 给依据 + 建设性挑战 > 礼貌但空洞 |
| 跳过的事 | `personal-workflow-map.md` 是空的（被建议但用户跳过了）；凭直觉挑了写作类切入 | 一次提醒过即可，不反复劝；用真实产出来反向证明 ROI |

## 3. 已达成的核心设计哲学

按重要性排序。这些是用户与 AI 在多轮对话中**已达成共识**的，不要重新论证：

1. **真实产物驱动 skill 设计**——凭空设计 meta-skill 是陷阱
2. **杠杆 ≠ 工具**：高 ROI skill = 频次 × 单次耗时 × 决策可复用度。低于一年省 10 小时的不值得造
3. **skill 是函数**：必须定义清楚 4 件事——适用条件 / 输入 / 输出（格式+评判标准）/ 处理原则
4. **单一职责**：不写大 skill，复杂任务用多个小 skill 组合调用
5. **skill 分级 L1-L5**：
   - L1 笼统角色 → 输出靠模型心情
   - L2 决策树 → if A then 1 / if B then 2
   - L3 输出标准 + 自检 → 不合格重写
   - L4 description 精确到自动触发 → 不用提示
   - L5 多 skill 自动编排 → agent 自决调度
6. **决策框架 skill 化，不是整件事 skill 化**——"工作太复杂无法 skill 化"是抽象层级错误的判断
7. **写作类 skill 必须区分阻力来源**：
   - "做的累" = 生产力问题，普通 skill 可解
   - "不愿开始" = 心理问题（完美主义），需要"分离生成与评估"型 skill
8. **voice profile 是关键资产**——一次性投入、长期复利；所有写作 skill 引用同一份
9. **AI 编造事实是 bug**——bones 没给具体时间 / 数字 / 比例时，用模糊词或反问用户，不要凭空填具体数字

## 4. skill 体系的"成熟度路径"

每个 skill 应清楚自己在追第几级。**别一上来就追 L4**——大多数 skill L2-L3 就够用。

| 级别 | 形态 | 用户体验 |
|------|------|----------|
| L1 | 提示词模板 | 输出后用户得改 |
| L2 | 决策树 | 常见分支走对路 |
| L3 | 闭环自检 | skill 内置"如何判断起效"，错了能自修 |
| L4 | 自动触发 | description 精确，agent 零提示自调 |
| L5 | 多 skill 自动编排 | agent 自决调度顺序 |

**L4 的关键不在 skill 主体，而在 description**——description 的边际收益远高于主体优化。

## 5. 与用户协作时的关键原则

### 当用户问框架性大问题时

不要直接给"7 步法"。流程：
1. 先看仓库现状（用户已经做了什么）
2. 反映给用户「我看到的」，让他感受到你真的看了
3. 提**视角转换**（杠杆/价值维度，不是工具维度）
4. 给**基于他现状的、有依据的**建议
5. 让他校准（AskQuestion 或 "你认这个吗"）

### 当用户给 skill 输入时

按 skill 自己的工作流**严格执行**。校验失败（如 bone 是空泛标签）→ 打回，但给补全方向。**不要替用户编内容**——这违反 skill 的核心约束，也会让用户失去对产物的认领感。

### 当用户给反馈时

- 用户说"像我"的句子 → 加入 voice profile 金句锚点
- 用户说"不像我"的句子 → 反向追查 voice profile 哪条规则失效
- 用户做事实校正 → 在 skill 反例里加对应条目
- 用户跳过了某步骤 → 一次提醒，不反复劝

### 当自己写的 skill 暴露 bug 时

**诚实承认**，不要假装"故意设计如此"。列 bug + 修复方向 + 标注是否阻塞当前任务。

### 何时该停下让用户决策

- 内容立意（thesis、bones、观点选择）
- 模式判定不确定时（A / B / mixed）
- 待造 skill 优先级
- 真实事实（时间、数据、产品名称）

### 何时不该问，直接做

- skill 内部工作流的执行细节
- voice profile 已覆盖的语言风格选择
- 修补 bug 的最小改动

### 接下来该做的（按优先级）

1. **不要立刻催下一篇** —— 让用户充分享受首篇发布的成就感，至少等他主动说"想写下一篇"或"想做别的"
2. 当用户回到 skill 工作时，问"你想做哪个"：
   - (a) 写下一篇博客（继续 draft-blog-post 跑通流程）
   - (b) 设计 `pre-publish-checklist`——**强烈推荐**：用户刚体验过"发布瞬间"，对发布焦虑的记忆最新鲜，此时设计 checklist 准度最高，且必须带"曝光度分级"维度
   - (c) 设计 `paragraph-triage`——可延后，首篇没分诊需求，下次某篇真要砍段时再造
   - (d) 转向工作类 skill（code review / tech review）——ROI 高，但跟写作 skill 是两条线，看用户精力
3. **如果用户问"接下来该写什么主题"**：可主动提议「我用了 skill 写出一篇关于 skill 的文章，这个递归瞬间本身就是下一篇的题」——这是会话里已铺过的种子

### 其他可能的反馈分支

| 用户反馈 | AI 下一步 |
|----------|-----------|
| "某几段我想改" | 用真实分诊需求设计 `paragraph-triage` |
| "下次写不动了"（重新阻力） | 排查：是 thesis 模糊还是发布焦虑？前者是 skill 输入问题，后者是 checklist 需求 |
| "voice profile 哪句不对" | 反向调 voice profile，新增反例 / 金句锚点 |
| "我想造 X 工作类 skill" | 进入 ROI 评估，参考第 5 节"框架性大问题"协作流程 |

### 中长期方向（按优先级）

1. 把写作 skill 体系（draft → triage → checklist → rewrite）补全到 L3-L4
2. 转向工作类 skill（`code-review-my-patterns`、`tech-review-checklist`）——这是用户工作流中最高 ROI 区域
3. 当 skill ≥ 5 个后，开始追 L4（description 自动触发）
4. 当 skill ≥ 8 个后，开始尝试 L5（skill 间自动编排）
5. 任何时候用户主动回到 `personal-workflow-map.md` 时——帮他填，重排 skill 造的优先级

## 8. 严禁的事（不要犯的协作错误）

- ❌ 替用户编 thesis / bones / 数据 / 时间 / 案例
- ❌ 给"看起来正确但没有深度"的框架式回答
- ❌ 在用户没要求时大量造文件（包括"顺手"造 meta-skill）
- ❌ 假装自己设计的 skill 完美无 bug
- ❌ 反复劝用户做他已经决定跳过的事
- ❌ 当用户的元思维已经很强时，给"skill 是什么"型科普
- ❌ skill description 用第一人称 / 缺 WHAT 或 WHEN
- ❌ 引用文件嵌套超过一级（a.md → b.md → c.md）
- ❌ 出现 voice 黑名单词（赋能 / 加持 / 颠覆 / 综上所述 等，详见 voice profile）

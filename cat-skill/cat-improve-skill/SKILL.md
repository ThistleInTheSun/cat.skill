# cat-improve-skill

改进 `cat-skill` 仓库中单个 skill 的**表现能力**：通过测试场景评测它是否能被准确触发、是否给 agent 足够的任务信息、是否能稳定产出高质量结果，并给出可执行改稿。

不做基础合规检查，不检测多个 skill 的功能重叠；这些场景分别交给 `cat-review-skill` 和 `cat-overlap-skills`。

## 输入

- `target`：被评测的 skill 目录路径，例如 `cat-skill/cat-create-skill/`
- `goal`（可选）：用户希望提升的方向，例如触发更准、步骤更稳、输出更可用、token 更省

## 工作流

复制并跟踪：

```
- [ ] 1. 读取目标 skill 的 SKILL.md 与同目录一级附属文件
- [ ] 2. 建立能力画像（触发、边界、输入、流程、输出、验证）
- [ ] 3. 设计 3-5 个代表性测试场景，并标注无 skill 基线
- [ ] 4. 按评分表评测表现
- [ ] 5. 诊断主要短板
- [ ] 6. 给出最小可执行改进方案
- [ ] 7. 用原测试场景复验改稿，并输出关键片段或完整 SKILL.md 草案
```

### Step 1 — 读取材料

读取目标 skill 的：

- `SKILL.md`
- 同目录一级附属文件，如 `reference.md`、`examples.md`
- 同目录 `scripts/` 下与工作流直接相关的脚本说明或入口文件

不要递归读取二级以上参考链，除非目标 skill 明确要求。

### Step 2 — 建立能力画像

提取并记录：

```
{
  "name": "<skill name>",
  "object": "<作用对象>",
  "action": "<核心动作>",
  "triggers": ["<触发词或触发场景>"],
  "non_goals": ["<明确不做什么>"],
  "inputs": ["<需要用户或仓库提供什么>"],
  "workflow": ["<关键步骤>"],
  "output": "<产物形态>",
  "verification": ["<如何判断起效>"]
}
```

### Step 3 — 设计测试场景

为目标 skill 设计 3-5 个场景，覆盖：

- **正向触发**：用户请求明确落在该 skill 范围内
- **边界触发**：请求与相邻 skill 接近，容易误用
- **缺参触发**：用户目标明确但信息不足
- **复杂输入**：目标文件较长、有附属文件，或需求包含多个约束
- **反向场景**：用户请求不应由该 skill 处理

每个场景都要补充：

- **无 skill 基线**：不使用该 skill 时，agent 大概率会怎样处理
- **期望增益**：使用该 skill 后，应在哪些方面明显更好

### Step 4 — 评分表

每项按 0-2 分评分：

| 维度 | 0 分 | 1 分 | 2 分 |
|------|------|------|------|
| 触发准确性 | 容易误触发或漏触发 | 触发词存在但边界含糊 | WHAT / WHEN / 非目标清晰 |
| 任务闭环 | 只描述原则 | 有步骤但缺少复验 | 输入、流程、输出、复验完整 |
| 操作稳定性 | 依赖临场发挥 | 有部分模板或清单 | 关键路径有清单、模板或评分标准 |
| 输出可用性 | 产物形态不明确 | 有产物但难直接执行 | 输出格式具体且可落地 |
| 效果增益 | 看不出比无 skill 更好 | 有增益但不稳定或不明确 | 测试场景中明显优于无 skill 基线 |
| 上下文效率 | 堆砌常识或重复内容 | 信息量可接受 | 只保留执行所需信息 |

总分解释：

- `0-4`：需要重写核心结构
- `5-8`：需要补齐关键环节
- `9-12`：只需局部增强

### Step 5 — 诊断短板

只诊断影响实际表现的问题，优先级如下：

1. 触发与边界：description 是否让 agent 选得准
2. 输入与缺参：是否知道何时直接做、何时提问
3. 工作流：步骤是否能把任务稳定推进到完成
4. 输出：结果是否有固定骨架，方便用户使用
5. 效果增益：测试场景中是否明显优于无 skill 基线
6. 复验：是否能判断改进后的 skill 已经起效
7. token 经济性：是否有可以删除的低价值内容

不要把基础合规项当作主要结论；若发现明显合规问题，只在“附带风险”中简短提示。

若改动风险高、评分争议大，或用户要求更严格验证，可使用子 agent 或第二轮干跑做独立评估；否则保持单轮轻量评测。

### Step 6 — 改进方案

每条改进建议必须包含：

- 问题：当前哪里影响能力表现
- 影响：它会导致怎样的误用、漏用或低质量输出
- 修改：可直接执行的改法
- 片段：必要时给出可替换文本

优先给出最小改动。只有当结构问题贯穿全篇时，才建议重写。

### Step 7 — 输出格式

使用以下报告骨架：

```markdown
# Improvement Report: <category>/<skill-name>

## 结论
- 总分：<N>/12
- 评级：<需要重写核心结构 / 需要补齐关键环节 / 只需局部增强>
- 主要短板：<一句话>

## 能力画像
<填写 Step 2 的画像>

## 测试场景
1. <场景> -> 无 skill 基线：<大概率表现>；期望增益：<使用 skill 后应变好之处>
2. <场景> -> 无 skill 基线：<大概率表现>；期望增益：<使用 skill 后应变好之处>
3. <场景> -> 无 skill 基线：<大概率表现>；期望增益：<使用 skill 后应变好之处>

## 评分明细
| 维度 | 分数 | 证据 |
|------|------|------|
| 触发准确性 | <0-2> | <证据> |
| 任务闭环 | <0-2> | <证据> |
| 操作稳定性 | <0-2> | <证据> |
| 输出可用性 | <0-2> | <证据> |
| 效果增益 | <0-2> | <证据> |
| 上下文效率 | <0-2> | <证据> |

## 改进建议
1. <问题 / 影响 / 修改 / 片段>
2. <问题 / 影响 / 修改 / 片段>
3. <问题 / 影响 / 修改 / 片段>

## 建议改稿
<给出关键替换片段；若用户要求落地修改，再编辑文件>

## 改后复验
- <用原测试场景复验，说明改稿在哪些维度有明确提升；若没有提升，说明不建议采用>

## 验证方式
- <如何用测试场景判断改进后已起效>
```

## 审阅原则

1. **能力优先**：重点评测实际任务表现，不重复 `cat-review-skill` 的硬规则清单
2. **证据驱动**：每个扣分点必须引用目标 skill 的具体内容或缺失项
3. **最小改动**：能用局部替换解决时，不建议整篇重写
4. **边界清晰**：遇到基础合规检查或重叠审计请求时，提示改用对应 skill
5. **可验证**：每个改进方案都要能被测试场景复验
6. **避免自证**：高风险或争议改稿可引入独立评估；普通改进不强制增加流程负担

## 验证本 skill 是否起效

执行后产物应满足：

- 报告中有 0-12 分总分与评分明细
- 至少包含 3 个代表性测试场景
- 测试场景包含无 skill 基线与期望增益
- 每条主要建议都包含问题、影响、修改和可执行片段
- 明确区分能力提升、基础合规检查和多 skill 重叠审计
- 给出改后复验结论和验证方式

## 反例

避免：

- 只说“建议更清晰、更精炼”，但不给具体替换片段
- 用合规 PASS/FAIL 报告替代能力评测
- 只评价文本质量，不设计测试场景
- 只给改稿，不用原测试场景复验是否真的变好
- 建议新增大量常识性说明，导致 skill 更难触发和使用
